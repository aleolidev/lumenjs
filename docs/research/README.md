# Product research

This directory maps the demonstrated needs of ambitious creature-RPG fangames
before LumenJS chooses an architecture.

The research has two separate subjects:

1. Games and ROM hacks show which player and creator capabilities matter.
2. Engines, tools, and repositories show how those capabilities are authored,
   implemented, tested, and distributed.

A project can be a useful product reference without being a safe code source.
Direct reuse requires a separate license and provenance review.

## Artifacts

- [method.md](method.md) defines evidence and evaluation rules.
- [capabilities.md](capabilities.md) is the evolving domain taxonomy.
- [corpus.md](corpus.md) records the initial reference corpus and why each
  project is included.
- [matrix.md](matrix.md) maps demonstrated capability families to initial
  product references.
- [findings.md](findings.md) records cross-project conclusions and antipatterns.
- [technical-sweep.md](technical-sweep.md) records the first repository sweep
  across rendering, engines, gameplay simulation, and authoring tools.
- [reference-map.md](reference-map.md) assigns preliminary technical references
  to LumenJS needs and records deliberate rejections.
- [capability-assignments.md](capability-assignments.md) is the consolidated
  scored assignment of references to product and engineering capabilities.
- [scoring.md](scoring.md) defines repeatable scoring, confidence, and decision
  thresholds for source-level repository audits.
- [spikes.md](spikes.md) records executable questions, results, and resulting
  decisions without promoting spike code into engine architecture.
- [`audits/`](audits/) contains source-level scorecards pinned to releases or
  revisions.
- [classification.md](classification.md) defines how capabilities will later
  be assigned to the core, contracts, modules, tooling, projects, or services.

These documents are inputs to product scoping, not a final specification.
